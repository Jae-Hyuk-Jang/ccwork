# Feature Planner

하나의 기능을 "아이디어"에서 "개발 가능한 이슈 단위"로 바꾸는 3단계 파이프라인이다. `docs/features/feature-planning-workflow.md`의 워크플로우를 그대로 실행하는 자체완결형 스킬이다 — 실행 중 그 문서를 별도로 다시 읽을 필요가 없다.

```
spec-original.md (초기 아이디어)
    ↓ 단계 1 — 요구사항 인터뷰
spec-fixed.md (확정된 요구사항)
    ↓ 단계 2 — PRD 작성 + 기술 결정(ADR)
prd.md (단일 기준 문서)
    ↓ 단계 3 — 이슈 분해
issues.md + (선택) GitHub Issues (실행 가능한 작업 단위)
```

**이 스킬의 존재 이유**: AI는 인터뷰·게이트 없이 곧장 코드를 작성하는 쪽으로 편향돼 있다. 이 스킬은 그 편향을 막기 위한 것이므로, 각 단계 사이의 `[GATE]`를 절대 건너뛰지 않는다 — 사용자 확인 없이 다음 단계로 넘어가면 이 스킬은 실패한 것이다.

## 입출력 계약

- **입력**: `$ARGUMENTS`로 기능 이름(`{name}`, 슬러그 형태— 예: `tag`, `search`)을 받는다. 인자가 없거나 슬러그로 쓰기 애매하면 사용자에게 되묻는다.
- **저장 위치**: 모든 산출물은 `docs/features/{name}/` 아래 저장한다. 디렉토리가 없으면 먼저 만든다.
- **최종 산출물**: `spec-fixed.md`, `prd.md`, `issues.md`, (선택) GitHub Issues.

## 시작하기 전에

- `docs/features/{name}/spec-original.md`가 이미 있으면 그걸 입력으로 쓴다. 없으면 사용자에게 초기 아이디어를 한두 문단으로 받아 그 내용으로 `spec-original.md`를 먼저 만든다.
- 이미 `spec-fixed.md`/`prd.md`/`issues.md` 중 일부가 존재하면, 처음부터 다시 하지 말고 있는 단계는 건너뛰어 다음 단계부터 진행한다 (예: `spec-fixed.md`가 이미 확정돼 있으면 단계 2부터 시작).
- 인터뷰/선택 질문은 **AskUserQuestion 도구로 한 번에 하나씩** 묻는다 — 여러 질문을 한 메시지에 몰아서 던지지 않는다. 추천 옵션엔 항상 이유를 함께 붙인다.

## 단계 1: 요구사항 인터뷰

**목적**: 초기 기능 정의서의 모호성을 제거하고, 엣지 케이스와 결정 사항을 확정한다.

**입력**: `docs/features/{name}/spec-original.md`, 코드베이스(기존 구조·패턴 참조).

**핵심 활동**: 정의서와 코드베이스를 함께 보고, 구현 전에 확인이 필요한 것들을 하나씩 질문한다. 다룰 영역(전부 물을 필요는 없다 — 스펙에 이미 답이 있는 항목은 건너뛴다):

1. 이 기능의 primary user는 누구인가?
2. 최소 동작 시나리오 3개는? (가장 기본적인 사용 흐름)
3. 데이터 저장 방식 — 기존 구조와의 관계는?
4. 경계 조건 — 최대값, 빈 값, 중복, 동시성은?
5. 에러 처리 — 실패 시 사용자에게 어떻게 표시하는가?
6. 기존 UI/코드 패턴과의 일관성 — 어떤 컴포넌트/모듈을 재사용할 수 있는가?
7. 성능 제약 — 허용 응답 시간, 데이터 크기 제한은?
8. 향후 확장 가능성 — 이 결정이 다음 기능에 영향을 주는가?

각 질문은 추천 방식과 이유를 함께 제시한다 (예: "trim + 대소문자 무시로 비교하는 걸 추천합니다 — 실수로 비슷한 값을 중복 생성하는 걸 막기 위해서입니다").

**Ubiquitous Language 정의**: 인터뷰 중 등장한 용어를 고정한다. 코드·문서·대화에서 같은 단어가 같은 것을 가리키게 하고, `spec-fixed.md`에 "용어 정의" 섹션으로 명시한다.

**산출물**: `docs/features/{name}/spec-fixed.md` — 원본 요구사항, 확정된 설계 결정(표: # / 항목 / 결정 / 근거), 용어 정의를 포함한다.

> **[GATE] 사용자가 `spec-fixed.md`를 읽고 확정할 때까지 다음 단계로 넘어가지 않는다.** 파일을 다 썼다고 자동으로 승인된 것이 아니다 — 반드시 사용자에게 확인을 구하고 명시적인 동의를 받는다.

## 단계 2: PRD 작성 + 기술 결정(ADR)

**목적**: 요구사항·기술 결정·범위를 하나의 문서로 통합해 "이 기능에 대해 궁금하면 여기만 보면 된다"는 단일 기준점을 만든다.

**입력**: `docs/features/{name}/spec-fixed.md`, `docs/features/{name}/design.md`(있으면), 코드베이스.

### 2-1. PRD 뼈대 생성 (기술 결정은 TODO로 비워둠)

PRD 구조 — 이 순서와 항목을 그대로 따른다:

- **개요** — 기능의 목적과 범위
- **사용자 스토리** — 사용자 관점의 흐름
- **기술 결정** — 어떤 구조 위에 올릴 것인가 (여기서 ADR 작성, 아래 2-3 참고)
- **Out of Scope** — 이번에 하지 않을 것
- **용어 정의** — `spec-fixed.md`의 Ubiquitous Language를 그대로 동기화

### 2-2. 아키텍처 3안 제안 & 비교

최소 3가지 구현 안을 스스로 제안하고, **아래 7가지 고정 기준**으로 비교한다 (기준을 임의로 바꾸거나 줄이지 않는다):

| #   | 기준                 | 설명                                           |
| --- | -------------------- | ---------------------------------------------- |
| 1   | 데이터 구조          | 어떤 형태로 데이터를 저장/표현하는가           |
| 2   | API 레이어 변경지점  | 기존 API에서 무엇이 바뀌는가                   |
| 3   | 상태관리 변경지점    | Context/Store에서 무엇이 추가되는가            |
| 4   | 핵심 동작            | 이 기능의 가장 중요한 동작이 어떻게 구현되는가 |
| 5   | 컴포넌트 구조        | 어떤 컴포넌트가 추가/수정되는가                |
| 6   | 기존 패턴과의 일관성 | 코드베이스의 기존 관습과 얼마나 맞는가         |
| 7   | 테스트 용이성        | 단위 테스트, E2E 테스트 작성이 쉬운가          |

비교 결과를 표로 정리해 사용자에게 제시한다.

> **[GATE] 사용자가 3안 중 하나를 선택할 때까지 다음 단계로 넘어가지 않는다.** 가장 나아 보이는 안을 추천할 수는 있지만, 임의로 골라서 확정하지 않는다.

### 2-3. 선택한 안을 ADR 4요소로 "기술 결정" 섹션에 작성

각 기술 결정은 아래 형식을 그대로 따른다:

```markdown
### 기술 결정 제목

**Context** — 이 결정이 필요한 맥락. 무엇을 풀고 있는가.
**Decision** — 무엇을 선택했는가.
**Alternatives** — 거부한 안과 각각의 거부 이유. (이유 없는 거부 금지)
**Consequences** — 트레이드오프. 장점뿐 아니라 단점도 반드시 명시한다.
```

`Alternatives`에 이유를 안 쓰거나, `Consequences`에 장점만 쓰는 건 이 형식을 지킨 게 아니다.

### 2-4. Out of Scope 작성

"하지 않을 것"을 구체적으로 나열한다. 이게 특히 중요한 이유: 명시하지 않으면 AI가 맥락을 확장 해석해 범위를 넘어서는 구현을 하게 된다 (예: 태그 기능을 만드는데 "태그 색상 커스터마이징", "태그 자동 추천", "태그 기반 검색"까지 만들어버리는 식). 스펙에 없는 인접 기능들을 적극적으로 찾아서 나열한다.

> **[GATE] 사용자가 Out of Scope을 읽고 확정할 때까지 다음 단계로 넘어가지 않는다.**

**산출물**: `docs/features/{name}/prd.md`

## 단계 3: 이슈 분해

**목적**: PRD를 실행 가능한 작업 단위(GitHub Issues)로 변환한다. 각 이슈는 독립적으로 TDD 사이클을 돌릴 수 있어야 한다.

**입력**: `docs/features/{name}/prd.md`, (선택) GitHub 프로젝트 보드.

### 수직 슬라이싱 원칙

이슈를 쪼갤 때마다 스스로에게 묻는다:

> "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"

- **Yes** → 수직 슬라이스다. TDD 사이클(Red→Green→Refactor)을 이 이슈 단위로 돌릴 수 있다.
- **No** → 다른 이슈와 합치거나 다시 나눈다.

**수평 슬라이싱(레이어별: API → 상태관리 → UI로 이슈를 나누는 것)은 금지한다.** 마지막 이슈까지 완료해야 눈에 보이는 동작이 나오면, 이슈 단위로 TDD를 돌릴 수 없다 — 이건 이 원칙의 실패 신호다.

### 이슈 크기

반나절~하루 안에 TDD 사이클을 완료할 수 있는 크기로 잡는다. 너무 크면 피드백 루프가 느려지고, 너무 작으면 PR 오버헤드가 실제 작업량을 초과한다.

### 의존성 순서

앞 이슈의 결과가 다음 이슈의 입력이 되도록 배치한다. 역방향 개발(뒤 이슈가 끝나야 앞 이슈가 의미 있어지는 순서)은 금지한다.

### Acceptance Criteria — Given-When-Then 형식

각 이슈는 반드시 아래 형식의 AC를 포함한다:

```markdown
## Acceptance Criteria

- [ ] Given [사전 조건], When [행동], Then [기대 결과]
- [ ] Given [사전 조건], When [행동], Then [기대 결과]
```

AC 없는 이슈는 만들지 않는다 — AC는 이후 테스트 시나리오의 씨앗이 된다.

> **[GATE] 사용자가 이슈 목록을 읽고 다음을 확인할 때까지 다음 단계(GitHub 등록)로 넘어가지 않는다:** 수직 슬라이스 기준 충족 여부, AC의 구체성, 의존성 순서.

**산출물**: `docs/features/{name}/issues.md` — 각 이슈는 제목/설명/완료 조건(AC)/Given-When-Then을 갖춘다.

### (선택) GitHub 등록

사용자가 원하면, 확정된 이슈를 GitHub Issues와 프로젝트 보드에 등록한다. `gh` CLI가 설치·인증돼 있는지 먼저 확인하고, 프로젝트 번호/오너를 모르면 사용자에게 묻거나 `gh project list`로 후보를 보여준다.

```bash
gh issue create --title "이슈 제목" --body "설명 + AC + 의존성"
gh project item-add <PROJECT_NUMBER> --owner <OWNER> --url <ISSUE_URL>
```

등록 후에는 각 이슈가 프로젝트 칸반 보드의 Todo 컬럼에 들어갔는지 확인한다.

## 승인 게이트 요약

| 지점        | 확인 내용                                         |
| ----------- | ------------------------------------------------- |
| 단계 1 후   | `spec-fixed.md` — 모호성 제거, 용어 확정          |
| 단계 2-2 후 | 3가지 안 비교 — 사용자가 안을 선택함              |
| 단계 2-4 후 | Out of Scope — 범위 확정                          |
| 단계 3 후   | 이슈 목록 — 수직 슬라이스, AC 구체성, 의존성 순서 |

이 게이트들을 생략하면 다음 단계 산출물이 사용자가 원치 않는 방향으로 굳어진 채 그대로 진행된다. 매 게이트마다 명시적으로 멈추고 확인을 구한다 — "이대로 진행할까요?"라고 묻는 것으로 충분하며, 답이 오기 전엔 다음 단계 파일을 쓰지 않는다.
