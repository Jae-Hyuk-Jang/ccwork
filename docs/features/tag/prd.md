# 태그 기능 PRD

`spec-original.md`(원본 요구사항)와 `spec-fixed.md`(질문/답변으로 확정한 설계 결정)를 기반으로 한 PRD 뼈대다. 세부 UI/데이터 결정의 근거는 `spec-fixed.md`를 참고한다.

## 1. 개요

노트에 태그를 추가·삭제하고, 노트 상세 화면에서 태그 목록을 확인할 수 있게 한다. 목적은 노트를 주제별로 가볍게 분류해 나중에 다시 찾기 쉽게 만드는 것이다 (`spec-original.md` 기준).

## 2. 사용자 스토리

| #   | 스토리                                                                                                                                      |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | 사용자로서, 노트 상세 화면에서 태그를 추가하고 싶다. 그래야 노트를 주제별로 분류해둘 수 있다.                                               |
| 2   | 사용자로서, 노트에 추가된 태그를 삭제하고 싶다. 그래야 더 이상 맞지 않는 분류를 정리할 수 있다.                                             |
| 3   | 사용자로서, 노트 상세 화면에서 이 노트에 붙은 태그 목록을 한눈에 확인하고 싶다. 그래야 이 노트가 어떤 주제로 분류돼 있는지 바로 알 수 있다. |
| 4   | 사용자로서, 같은 태그를 대소문자만 다르게 실수로 여러 번 추가해도 하나로 처리되길 원한다. 그래야 태그 목록이 지저분해지지 않는다.           |
| 5   | 사용자로서, 노트를 저장하지 않고 취소하면 태그 변경도 함께 취소되길 원한다. 그래야 실수로 되돌릴 수 없는 변경을 만들지 않는다.              |

## 3. 기술 결정 (ADR)

> `Context` / `Decision` / `Alternatives` / `Consequences` 형식을 따른다. `spec-fixed.md`의 결정 사항·근거와 아키텍처 대안 비교를 바탕으로 채웠다.

### ADR-001: 태그 데이터 모델 및 저장 방식

- **Context**: 노트에 태그를 추가/삭제/조회하는 기능을 구현해야 한다. 태그를 `Note`와 어떻게 연결할지(내장 vs 정규화) 결정이 필요하다. 현재 스펙엔 태그 검색·자동완성·노트 간 공유 요구사항이 없다 (4. Out of Scope 참고).
- **Decision**: `Note` 타입에 `tags?: string[]` 필드를 그대로 내장한다. 별도 태그 엔티티/API/Context를 만들지 않는다.
- **Alternatives**: 정규화된 태그 레지스트리 — `Note`에서 `tags`를 분리해 별도 `TagsContext`/`api/tags.ts`로 관리(`tagsByNoteId` 맵 또는 전역 태그 목록). 태그 검색/자동완성/노트 간 공유가 필요해지면 유리하지만, 현재 스펙엔 해당 요구사항이 전혀 없어 기각(YAGNI).
- **Consequences**: (+) `api/notes.ts`의 `createNote`(`Omit<Note,...>`)/`updateNote`(`Partial<Note>`)를 변경 없이 그대로 재사용해 `tags`를 통과시킬 수 있다. 새 API 파일/Context가 불필요하다. (–) 이후 태그 검색·자동완성·노트 간 통계 같은 기능이 필요해지면 이 시점에서 정규화 리팩터링이 필요해진다(마이그레이션 비용을 뒤로 미루는 선택).

### ADR-002: 태그 저장 시점 (배치 저장 vs 즉시 API 반영)

- **Context**: 태그 추가/삭제 시점에 서버에 바로 반영할지, 제목/내용처럼 저장 버튼을 눌러야 반영할지 결정이 필요하다. 이 프로젝트는 이미 제목/내용을 로컬 state로 들고 있다가 저장 시점에만 API를 호출하는 기존 패턴(`NoteEditor`)을 갖고 있다.
- **Decision**: 태그도 로컬 state(`NoteEditor`의 `tags`)로만 관리하다가, **'저장' 버튼 클릭 시** `createNote`/`updateNote` 호출과 함께 제목/내용과 한 번에 반영한다(배치 저장). 태그 추가/삭제마다 즉시 API를 호출하지 않는다.
- **Alternatives**: 태그 즉시 반영 — `NotesContext`에 `addTagToNote`/`removeTagFromNote` 액션을 추가해 칩 조작 즉시 API 호출. 기각 사유: '취소' 버튼을 눌러도 이미 서버에 반영된 태그는 되돌릴 수 없어 스펙("취소 시 태그 변경도 함께 되돌아감")을 위반하고, 새 노트 작성 중(아직 id 없음)엔 이 액션 자체가 호출 불가능해 별도 분기가 필요해진다.
- **Consequences**: (+) 제목/내용과 완전히 동일한 저장 정책이라 사용자가 "저장을 눌러야 반영된다"는 멘탈 모델 하나만 학습하면 된다. '취소'가 태그 변경까지 포함해 정확히 전체 롤백된다. (–) 저장을 누르기 전까지는 다른 화면(사이드바 목록)에 태그 변경이 반영되지 않는다. 여러 필드를 한 번에 묶어 보내므로, 태그만 따로 실패했을 때의 부분 실패 처리는 불가능하다(현재는 전체 성공/실패만 존재).

### ADR-003: 태그 중복 판정 로직

- **Context**: 스펙은 "중복 태그는 허용하지 않는다"고만 되어 있고 대소문자 처리 기준이 없다. 사용자가 'React'를 입력한 뒤 실수로 'react'를 또 입력하는 경우를 어떻게 처리할지 결정이 필요하다.
- **Decision**: 태그 문자열은 앞뒤 공백을 trim하고 **대소문자를 무시**해 비교한다 — 'React'와 'react'는 동일 태그로 간주해 중복 추가를 막는다. 단, 저장/표시는 사용자가 입력한 원문(대소문자 그대로)을 유지한다.
- **Alternatives**: 대소문자까지 구분하는 strict 비교 — 'React'와 'react'를 서로 다른 태그로 허용. 기각 사유: 기술 용어처럼 대소문자가 의미를 가지는 태그를 구분하고 싶을 때는 유리하지만, 사용자가 실수로 비슷한 태그를 중복 생성하기 쉬워 스펙의 "중복 방지" 의도에 어긋난다.
- **Consequences**: (+) 실수로 비슷한 케이스의 태그가 중복 생성되는 것을 막아 태그 목록이 깔끔하게 유지된다. (–) 의도적으로 대소문자를 다르게 쓰고 싶은 경우(드물지만)를 지원하지 못한다. 비교 로직(`NoteEditor.handleAddTag`의 `toLowerCase()` 비교)이 이후 정렬/검색 기능에서도 일관되게 적용돼야 한다(현재는 이 함수 안에만 존재).

### ADR-004: 컴포넌트 구조 — 커스텀 훅 + 합성

- **Context**: 현재 구현은 `tags`/`tagInput` state와 `handleAddTag`/`handleDeleteTag`/`handleCancel`의 리셋 로직이 모두 `NoteEditor` 컴포넌트 안에 직접 박혀 있다. 이 로직만 단위 테스트하려 해도 `NotesContext`를 mock하고 컴포넌트 전체를 렌더해야 한다. 앞으로 다른 컴포넌트에서도 커스텀 훅 활용을 늘려가고자 한다.
- **Decision**: 태그 상태/로직을 `useTags(initialTags)` 커스텀 훅으로 분리한다. 훅은 `{ tags, tagInput, setTagInput, addTag, deleteTag, reset }`을 반환하며, trim+대소문자 무시 중복 검사(ADR-003)와 리셋(취소/노트 전환 시 동기화, ADR-002) 로직을 캡슐화한다. 태그 입력창+칩 목록 렌더링은 `TagInputField`라는 프레젠테이션 컴포넌트로 빼서, `NoteEditor`는 `useTags()`를 호출해 `TagInputField`에 값을 넘겨주는 **합성(composition)**만 담당한다. `NoteEditor`는 title/content 로컬 state는 그대로 유지한다(ADR-001/002의 데이터 모델·저장 시점 결정 자체는 바뀌지 않음).
- **Alternatives**: 현재처럼 `NoteEditor`에 인라인으로 유지. 기각 사유는 아니고 트레이드오프 — 지금 이 기능 하나만 보면 파일이 1개(`NoteEditor.tsx`)에서 3개(`useTags.ts`, `TagInputField.tsx`, `NoteEditor.tsx`)로 늘어나는 간접화 비용이 있다. 다만 커스텀 훅 활용을 다른 컴포넌트로도 확장할 계획이라, 이번이 그 패턴의 첫 적용 사례가 된다.
- **Consequences**: (+) `useTags`를 `@testing-library/react`의 `renderHook`으로 컴포넌트 마운트 없이 단위 테스트할 수 있어, 중복 판정/리셋 로직 검증이 훨씬 빠르고 좁아진다. `TagInputField`는 `NotesContext` 없이 props만으로 독립 테스트 가능. `NoteEditor` 테스트는 "저장/취소 시 올바른 tags를 넘기는가" 같은 통합 관점만 남아 테스트 범위가 명확히 나뉜다. (–) 작은 기능 대비 파일/추상화 계층이 늘어나 처음 코드를 읽는 사람은 흐름을 따라가려면 두 파일을 더 열어봐야 한다.

## 4. Out of Scope

- 태그 기반 노트 검색/필터링
- 태그 이름 변경(rename) — 기존 태그를 수정하려면 삭제 후 재입력해야 한다
- 태그별 색상/아이콘 커스터마이징
- 태그 자동완성/추천
- 사이드바 목록(`NoteItem`)에서의 태그 추가/삭제 — 목록은 읽기 전용 표시만 지원한다 (`spec-fixed.md` 결정 #1)
- 태그 개수/글자수 제한 — 의도적으로 두지 않는다 (`spec-fixed.md` 결정 #4)

## 5. 용어 정의

| 용어            | 정의                                                                    |
| --------------- | ----------------------------------------------------------------------- |
| 태그 (Tag)      | 노트에 붙이는 자유 형식 문자열 라벨. `Note.tags?: string[]`로 저장한다. |
| 태그 칩 / 뱃지  | 태그를 화면에 표시하는 pill 모양 UI 요소 (`NoteTagBadge` 컴포넌트).     |
| 중복 태그       | 앞뒤 공백을 trim하고 대소문자를 무시했을 때 문자열이 동일한 태그.       |
| 상세 화면       | 노트를 보거나 편집하는 화면 (`NoteEditor` 컴포넌트).                    |
| 목록 / 사이드바 | 노트 목록을 보여주는 화면 (`NoteList`/`NoteItem` 컴포넌트).             |
