# 검색 기능 정의서 (확정)

`spec-original.md`의 기본 요구사항에 대해, 검색 대상·UI 패턴·매칭 규칙·엣지 케이스를 질문/답변으로 확정한 버전이다.

## 기능 개요

노트의 제목과 본문 내용을 검색할 수 있다.

## 기능 요구사항 (원본)

- 사용자가 검색어를 입력하면 본문 내용에 검색어가 포함된 노트를 찾을 수 있다.

## 확정된 설계 결정

| #   | 항목               | 결정                                                                                                                                                                                                                                                                          | 근거                                                                                                                                                                                                                                                                                                                                                                                                              |
| --- | ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | 검색 대상 범위     | **제목(title) + 본문(content)** 둘 다 검색 대상으로 삼는다. 태그는 제외한다.                                                                                                                                                                                                  | "검색"이라고 하면 사용자는 보통 제목도 당연히 포함될 거라 기대한다. 태그 검색/필터링은 별도 기능(이미 태그 칩이 있어 혼동 여지)이라 이번 범위에서 뺀다.                                                                                                                                                                                                                                                           |
| 2   | 검색 UI 위치       | 사이드바(`NoteList`) 상단에 검색창을 둔다.                                                                                                                                                                                                                                    | 이미 사이드바가 "노트 목록을 보여주는" 화면이므로, 그 목록을 바로 그 자리에서 필터링하는 것이 가장 직관적이다.                                                                                                                                                                                                                                                                                                    |
| 3   | 결과 표시 방식     | 별도 결과 화면/드롭다운 없이, **사이드바 목록(`NoteItem`) 자체를 필터링**해서 보여준다.                                                                                                                                                                                       | 새 UI 패턴(드롭다운, 결과 페이지)을 추가하지 않고 기존 `NoteList` 렌더링을 그대로 재사용할 수 있다.                                                                                                                                                                                                                                                                                                               |
| 4   | 검색 트리거        | 검색어를 입력할 때마다 **실시간으로** 목록이 필터링된다 (별도 버튼/Enter 없음).                                                                                                                                                                                               | 모든 노트가 이미 `NotesContext`에 로드돼 있어 클라이언트 필터링이므로 서버 요청 비용이 없다. 실시간 필터링이 검색창형 UI의 자연스러운 기대 동작이다.                                                                                                                                                                                                                                                              |
| 5   | 매칭 규칙          | 검색어와 제목/본문을 비교할 때 **앞뒤 공백을 trim하고, 대소문자를 무시하며, 부분 문자열(substring) 포함 여부**로 판정한다. **trim한 결과가 빈 문자열이면 "매치 없음"이 아니라 매칭 자체를 건너뛰고 전체 노트를 그대로 보여준다** (공백만 입력해도 빈 검색어와 동일하게 취급). | 대소문자 무시는 태그 중복 판정(ADR-003)과 일관된 정책이다. 부분 문자열 포함이 검색 기능의 최소 기대치다. 빈 문자열은 모든 문자열의 부분 문자열이라(`"abc".includes("")`가 `true`) 매칭 로직을 구현에 따라 그대로 태우면 의도치 않은 전건 일치나 무한 루프(인덱스 탐색 방식일 경우) 위험이 있어 별도 분기로 명시한다.                                                                                              |
| 6   | 선택된 노트 처리   | 검색으로 목록에서 필터링되어 사라진 노트라도, 그 노트가 현재 `NoteEditor`에 **선택되어 있다면 에디터는 그대로 유지**된다 (강제로 선택 해제하지 않는다).                                                                                                                       | `selectedNoteId`(선택 상태)와 검색어(목록 필터 상태)는 서로 독립적인 UI 상태다. 검색은 "무엇을 보여줄지"만 바꿀 뿐, "무엇이 편집 중인지"에는 관여하지 않는다.                                                                                                                                                                                                                                                     |
| 7   | 검색어 초기화      | 검색 입력창 안에 **x 버튼**을 두어, 값이 있을 때만 나타나고 클릭 한 번으로 검색어를 비운다.                                                                                                                                                                                   | 태그 입력창과 달리 검색창은 "필터 값 하나"만 가지므로, 값을 비우는 전용 UI가 있는 편이 사용성이 좋다.                                                                                                                                                                                                                                                                                                             |
| 8   | 매칭 하이라이트    | 목록에 표시되는 제목/본문 미리보기에서 검색어와 일치하는 부분을 하이라이트(강조 표시)한다. 배경색은 **`bg-tertiary-container/40`**(액센트 컬러 40% 불투명도) 토큰을 쓰고, 텍스트 색상은 원래 색(`on-surface`/`on-surface-variant`)을 그대로 유지한다.                         | 여러 노트 중 "왜 이 노트가 검색됐는지"를 한눈에 확인할 수 있게 한다. 색상은 디자인 시스템 자동 검사 훅(`.claude/hooks/check-design-principles.cjs`)이 하드코딩 색상을 금지하므로 반드시 기존 토큰을 써야 하고, `bg-surface-container-highest`(선택된 노트 배경과 동일 톤)를 쓰면 선택된 노트 안에서 하이라이트가 안 보이는 충돌이 생겨 액센트 컬러(`tertiary-container`)를 낮은 불투명도로 아껴 쓰는 쪽을 택했다. |
| 9   | 결과 없음 표시     | 검색어가 있고(빈 문자열이 아니고) 일치하는 노트가 0건이면, 기존 "노트가 없습니다"(노트 자체가 0건일 때 문구)와는 **다른 문구**("검색 결과가 없습니다" 등)를 보여준다.                                                                                                         | `NoteList`가 이미 loading/error/empty 3가지 상태를 구분해 보여주는 기존 패턴과 일관되게, "노트가 아예 없음"과 "검색 결과가 없음"을 구분해야 사용자가 헷갈리지 않는다.                                                                                                                                                                                                                                             |
| 10  | 개수/성능 제약     | 검색어 길이, 노트 개수에 **제한을 두지 않는다**. 클라이언트 메모리에 이미 로드된 `notes` 배열을 그대로 필터링한다 (서버 쿼리 없음).                                                                                                                                           | 이 앱은 소규모 개인 노트 앱이라 클라이언트 필터링으로 충분하다. 스펙에 없는 성능 제약을 임의로 추가하지 않는다.                                                                                                                                                                                                                                                                                                   |
| 11  | 검색창 접근성 라벨 | `SearchInput`의 `<input>`은 **`aria-label="노트 검색"`**을 가진다.                                                                                                                                                                                                            | `TagInputField`가 `aria-label="태그 입력"`을 쓰는 네이밍 관례를 따른다. `.claude/hooks/check-design-principles.cjs`가 모든 `<input>`에 `aria-label`(또는 `-ledby`)을 강제한다.                                                                                                                                                                                                                                    |
| 12  | 검색창 노출 조건   | 노트가 **애초에 0건**이거나 `NotesContext`가 **로딩 중/에러 상태**일 때는 검색창 자체를 보여주지 않는다. 목록이 실제로 렌더링되는 분기(노트가 1건 이상 있을 때)에서만 검색창이 나타난다.                                                                                      | 노트가 없거나 아직 안 불러와졌으면 검색할 대상 자체가 없다. 기존 `NoteList`의 loading → error → empty → 목록 순 early-return 구조를 그대로 유지하고, 검색창은 마지막 "목록" 분기 안에만 추가한다.                                                                                                                                                                                                                 |

## 용어 정의 (Ubiquitous Language)

| 용어           | 정의                                                                      |
| -------------- | ------------------------------------------------------------------------- |
| 검색어 (Query) | 사용자가 검색창에 입력한 문자열.                                          |
| 검색 대상      | 검색어와 비교하는 노트의 필드. 이 기능에서는 `title`, `content` 두 필드.  |
| 일치 (Match)   | 검색 대상 필드에 검색어가 대소문자 무시 상태로 부분 문자열로 포함된 경우. |
| 하이라이트     | 목록 미리보기에서 일치한 부분을 시각적으로 강조 표시하는 것.              |
| 검색창         | 사이드바 상단에 위치한, 검색어를 입력하는 텍스트 입력 UI.                 |

## 비고

- 검색은 `NoteEditor`(상세 화면)의 편집 상태(`selectedNoteId`, `isCreating`)와 무관하게 항상 사이드바에서 동작한다 — "새 노트" 작성 중에도 검색창은 그대로 사용 가능하다.
- 태그 검색/필터링(`#React` 클릭 시 필터링 등)은 이번 스펙에 포함하지 않는다 (결정 #1).
