# 검색 기능 이슈 분해

`prd.md`를 수직 슬라이싱(vertical slicing) 원칙으로 쪼갠 이슈 목록이다. 각 이슈는 레이어(훅/컴포넌트)별로 나뉘지 않고, 사용자에게 실제로 확인 가능한 가치를 end-to-end로 전달하는 단위로 나눴다. 순서대로 진행하면 각 이슈가 이전 이슈 위에서 독립적으로 데모 가능하다.

---

## 이슈 1: 사이드바 검색으로 노트 목록 실시간 필터링

**설명**

사이드바(`NoteList`) 상단에 검색창(`SearchInput`)을 추가한다. 검색어를 입력할 때마다 실시간으로, 제목 또는 본문에 검색어가 포함된 노트만 목록에 남는다. 매칭은 앞뒤 공백 trim + 대소문자 무시 + 부분 문자열 포함 기준이다(`useNoteSearch` 훅 + `findMatchRanges` 유틸, ADR-001/002). 이 단계에서는 하이라이트 없이 필터링만 동작한다.

**완료 조건 (Acceptance Criteria)**

기본 동작:

- [ ] Given 노트 목록에 "React 공부", "TypeScript 정리", "회의록" 노트가 있을 때, When 검색창에 "react"를 입력하면, Then "React 공부" 노트만 목록에 남는다 (대소문자 무시).
- [ ] Given 위와 같은 목록일 때, When 검색창에 "react"를 입력하면, Then "TypeScript 정리"와 "회의록"은 목록에서 완전히 사라진다 (일부만 흐리게 남거나 하지 않는다).
- [ ] Given 어떤 노트의 본문에 "타입스크립트"가 포함돼 있을 때, When 검색창에 " 타입 "(앞뒤 공백 포함)을 입력하면, Then 그 노트가 결과에 포함된다 (trim + 부분 문자열 포함).
- [ ] Given 검색어와 일치하는 노트가 하나도 없을 때, When 목록을 보면, Then 노트가 0건일 때와 다른 문구("검색 결과가 없습니다")가 표시된다.
- [ ] Given 검색창에 값이 입력된 상태일 때, When 검색창 안의 x 버튼을 클릭하면, Then 검색어가 비워지고 전체 노트 목록이 다시 보인다.
- [ ] Given 어떤 노트가 `NoteEditor`에 선택되어 열려 있을 때, When 검색어를 입력해 그 노트가 목록에서 필터링되어 사라지면, Then `NoteEditor`는 계속 그 노트를 그대로 보여준다.
- [ ] Given 검색창이 비어 있을 때, When 목록을 보면, Then 필터링 없이 전체 노트가 보인다.

빈 쿼리 / 경계값 (`findMatchRanges`, `useNoteSearch` 단위 테스트로 검증 — ADR-002):

- [ ] Given 검색창에 공백만(예: `"   "`) 입력했을 때, When 목록을 보면, Then trim 결과가 빈 문자열이므로 검색어가 비어있을 때와 동일하게 전체 노트가 필터링 없이 보인다.
- [ ] Given `findMatchRanges("아무 텍스트", "")`를 직접 호출했을 때, When 결과를 확인하면, Then 빈 배열(`[]`)을 즉시 반환하고 무한 루프나 예외 없이 종료된다.
- [ ] Given 검색어에 정규식 특수문자(예: `"."`, `"*"`, `"("`)가 포함돼 있을 때, When 검색하면, Then 정규식 패턴이 아니라 **리터럴 문자열**로 취급되어 매칭되고 에러가 발생하지 않는다.

노출 조건 (`spec-fixed.md` 결정 #12):

- [ ] Given 노트가 애초에 0건일 때, When 사이드바를 보면, Then 검색창 자체가 보이지 않고 기존 "노트가 없습니다" 문구만 표시된다.
- [ ] Given `NotesContext`가 로딩 중이거나 에러 상태일 때, When 사이드바를 보면, Then 검색창이 보이지 않는다.

접근성 (`spec-fixed.md` 결정 #11):

- [ ] Given 검색창이 렌더링되었을 때, When `aria-label`로 요소를 조회하면, Then `"노트 검색"` 라벨로 검색창 input을 찾을 수 있다.

## 이슈 2: 검색 결과 하이라이트 표시

**설명**

이슈 1에서 필터링된 목록의 제목/본문 미리보기에서, 검색어와 일치하는 부분을 `<mark>`로 강조 표시한다. 매칭 위치 탐색은 이슈 1에서 만든 `findMatchRanges` 유틸을 그대로 재사용한다(ADR-002) — 필터링 기준과 하이라이트 기준이 항상 같은 함수에서 나온다.

**완료 조건**

- [ ] Given 검색어 "react"를 입력했고 어떤 노트의 제목이 "React 공부"일 때, When 목록을 보면, Then 제목 중 "React" 부분이 강조 표시된다.
- [ ] Given 검색어 "타입"을 입력했고 어떤 노트의 본문에 "타입스크립트를 공부"가 포함돼 있을 때, When 목록을 보면, Then 본문 미리보기 중 "타입" 부분이 강조 표시된다.
- [ ] Given 검색어가 노트 제목과 본문 양쪽에 모두 일치할 때, When 목록을 보면, Then 제목과 본문 미리보기 **둘 다** 하이라이트된다.
- [ ] Given 같은 검색어가 본문 한 곳에 여러 번 등장할 때, When 목록을 보면, Then 일치하는 부분이 **모두** 하이라이트된다 (첫 번째 일치만 되는 게 아니다).
- [ ] Given 검색창이 비어 있을 때, When 목록을 보면, Then 어떤 노트 제목/본문에도 하이라이트가 적용되지 않는다.
- [ ] Given 검색어 "REACT"를 입력했고 본문 원문은 소문자 "react"일 때, When 목록을 보면, Then 원문 대소문자("react") 그대로 유지된 채 그 부분만 강조 표시된다 (원문 변형 없음).
- [ ] Given 검색어가 입력되어 하이라이트가 보이는 상태일 때, When 검색창 x 버튼을 눌러 검색어를 지우면, Then 하이라이트도 함께 사라진다 (필터링 해제와 하이라이트 해제가 함께 일어난다).
- [ ] Given 하이라이트가 렌더링되었을 때, When 하이라이트 요소의 클래스를 확인하면, Then 하드코딩된 색상이 아니라 디자인 토큰 클래스(`bg-tertiary-container/40`)가 적용돼 있다 (`.claude/hooks/check-design-principles.cjs` 색상 규칙 준수 확인).

## 참고

- 각 이슈는 `docs/features/search/prd.md`의 사용자 스토리/ADR과 대응한다: 이슈 1 ↔ 스토리 1·3·4·5·ADR-001, 이슈 2 ↔ 스토리 2·ADR-002.
- 이슈 2는 이슈 1이 만든 `findMatchRanges` 유틸과 `filteredNotes`/`query`를 그대로 재사용하는 의존 관계다 — 이슈 1 없이는 진행할 수 없다.
- Out of Scope(태그 검색, 정렬 변경, 검색 히스토리, 자동완성, 서버 사이드 검색, 정규식/고급 연산자, 단축키)에 해당하는 이슈는 만들지 않았다.
- 빈 쿼리/노출 조건/접근성 라벨(결정 #5, #11, #12)은 이슈 1에, 하이라이트 색상 토큰과 다중 일치(결정 #8)는 이슈 2에 추가했다 — `spec-fixed.md`·`prd.md` ADR-001/002 보강분과 1:1로 대응한다.

## 테스트 전략 메모

- `findMatchRanges`(순수 함수)는 `src/utils/textMatch.test.ts`에서 훅/컴포넌트 마운트 없이 바로 단위 테스트한다 — 빈 문자열, 공백만 있는 문자열, 대소문자 혼용, 정규식 특수문자, 다중 일치 위치를 표 기반(`it.each`)으로 촘촘히 검증하면 이슈 1/2의 경계값 AC 대부분을 여기서 커버할 수 있다.
- `useNoteSearch`는 `useTags`와 동일하게 `renderHook`으로 단위 테스트한다 (빈 쿼리 시 `notes` 그대로 반환하는지가 핵심 케이스).
- `SearchInput`은 `aria-label="노트 검색"`(결정 #11) 덕분에 RTL의 `getByLabelText`로 독립 테스트 가능하다 (`TagInputField` 테스트와 동일한 패턴).
- `NoteList`/`NoteItem` 통합 테스트는 "필터링된 목록이 올바르게 보이는가"와 "하이라이트 마크업이 올바른 위치에 붙는가"만 검증하면 되고, 매칭 규칙 자체의 경계값 재검증은 하지 않는다(중복 회피).
